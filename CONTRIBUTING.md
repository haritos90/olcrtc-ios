# Contributing to olcrtc-ios

Thanks for contributing. This guide covers the conventions this project expects.
Keep changes small and focused, and stay consistent with the code already in the
file you're editing.

---

## Language: English everywhere

**The entire repository is in English — no exceptions.** This keeps the codebase
accessible to every contributor and consistent with upstream
[olcrtc](https://github.com/openlibrecommunity/olcrtc).

| Where | Rule |
|---|---|
| **Code comments** | All `//` and `///` comments in English. |
| **Documentation** | README, `docs/`, this file, `TODO.md`, doc-comments — English. |
| **Commit messages** | English, Conventional Commits format (see below). |
| **Identifiers** | Types, functions, variables, enum cases — English. |
| **User-facing strings** | Authored in English as the source language, then localized. |

User-facing text is never hardcoded: add an `L10n` case and its entries in
`App/Localization/L10nTable.swift`. English (`en`) is the source language; every
other language is a translation of the English string.

Route **every** user-facing term through `L10n` — even ones deliberately kept in
English (brand/technical names, abbreviations). Add the case and give the other
languages an *explicit* entry equal to the English value rather than leaving a
literal in the view, so a future locale can translate it without hunting for
hardcoded strings. Identifiers shown to users (carrier/transport IDs, the default
group name) get a display-name mapping rather than rendering the raw token.

(Discussion with maintainers can be in any language — but anything committed to
the repo is English.)

## Brand name casing

The project name has two intentional forms, matching upstream
[olcrtc](https://github.com/openlibrecommunity/olcrtc). Pick by context:

| Form | Use it for |
|---|---|
| **`OlcRTC`** | The product name in **display & prose** — UI titles, the app's home-screen name (`CFBundleDisplayName`), README/docs sentences ("OlcRTC tunnels traffic…"). |
| **`olcrtc`** | **Technical identifiers** — the `olcrtc://` URL scheme, bundle id, container/file names, the repo slug `olcrtc-ios`, package paths. Swift **types** use the `Olcrtc` PascalCase prefix (`OlcrtcConnection`, `OlcrtcURI`, `OlcrtcApp`). |
| **`OLCRTC_`** | Server **env-var prefix** only (`OLCRTC_CARRIER`, …) — a wire contract with `srv.sh`; never re-case these. |

Don't introduce other spellings (`OLCRTC` as a word, `Olcrtc` in prose, `olcRTC`).

## Commit messages — Conventional Commits

Follow the [Conventional Commits](https://www.conventionalcommits.org/) spec,
matching upstream olcrtc:

```
<type>(<optional scope>): <short imperative summary>
```

Allowed types: `feat`, `fix`, `refactor`, `perf`, `docs`, `test`, `build`,
`chore`, `style`.

Examples:

```
feat(jitsi): add OLCRTC_JITSI_URL env var support
fix: strip whitespace from room ID on paste
refactor: remove jazz carrier from CarrierTransportMatrix
docs: add contributing guidelines
test: cover full-URL room ID round-trip
```

- Summary in the **imperative mood** ("add", not "added" / "adds").
- Keep it under ~70 characters; put detail in the body if needed.
- English only.

## Build & versioning

The Xcode project is generated by
[XcodeGen](https://github.com/yonaskolb/XcodeGen) from `project.yml` —
`olcrtc-ios.xcodeproj` is **not** tracked in git.

- After editing `project.yml`, run `xcodegen generate --spec project.yml`.
- **Build number** (`CFBundleVersion` in `project.yml`): bump by **1 per build**.
  It identifies which build is installed on a device. There is no auto-bump build
  phase — bump it by hand, **once per build, by whoever runs the build**, so
  concurrent agents never race the counter (see [AGENTS.md](AGENTS.md) §4).
- **Marketing version** (`MARKETING_VERSION` in `project.yml`): change only on a
  deliberate release decision, never automatically.
- `App/Info.plist` is **generated** from `project.yml` `info.properties` — never
  hand-edit it; `xcodegen` overwrites it.

Doc / comment / localization-string-only changes do **not** need a build-number
bump.

## Tests

Run the suite before opening a PR; all tests must pass:

```bash
# simulator names drift between Xcode versions — pick whatever iPhone is installed
UDID=$(xcrun simctl list devices available | grep -m1 'iPhone' | grep -oE '[0-9A-Fa-f-]{36}')
xcodebuild test \
  -project olcrtc-ios.xcodeproj \
  -scheme olcrtc-ios-tests \
  -destination "id=$UDID"
```

Add tests for new parsing/validation logic and for any change to the
server↔client wire contract (URI format, env vars, `srv.sh` patches).

## srv.sh parity

`scripts/srv.sh` is a verbatim copy of upstream `olcrtc-upstream/script/srv.sh` with our
changes wrapped in `# boc olcrtc-ios` / `# eoc olcrtc-ios` markers.
`scripts/parity_check.py` runs as an Xcode pre-build phase and fails the build if
any line **outside** those markers drifts from upstream. After touching
`srv.sh`, run it directly — it must pass:

```bash
python3 scripts/parity_check.py
```

See [How srv.sh works](README.md#how-srvsh-works) for the full patching workflow.

## Task tracking

Work is tracked in `TODO.md`, which flows `Backlog → Open → Closed`. Its **How
this file works** section is the source of truth for the lifecycle, columns, and
priority scheme — don't restate it here. In short: new tasks start in Backlog with
an optional detail block; closing a task moves its row to Closed, fills the
**Resolution** column (how it was resolved, or `Won't Do` / `Duplicate`), and
deletes its detail block.

---

## Marking task-driven changes

When you change code to implement a numbered `TODO.md` task, leave an inline trail
tying the change back to that task. Commits are coarse-grained — several tasks
often land in one commit — so these comments, not git history, are the per-task
change record the code itself carries: *what* changed, *for which task*, and *why*.

- **New or changed code** — tag it with the task ID and a short description:

  ```swift
  // #NNN: <one-line description of what this does / why>
  func newThing() { ... }
  ```

  (The #234 work, for example, tags its new code with `(#234)` throughout
  `TunnelManager`, `ConnectionsView`, and `PortAvailability`.)

- **Replaced code** — don't just delete it. Comment out the original directly
  above the replacement, tagged with the same task ID and `was:`, so the previous
  behaviour stays visible:

  ```swift
  // #NNN was: <what the old code did>
  // let old = legacyValue
  let new = improvedValue
  ```

- **Multi-line blocks** — wrap them with the same `boc` / `eoc` markers already
  used for `srv.sh`, carrying the task ID:

  ```swift
  // boc #NNN
  ... new code ...
  // eoc #NNN
  ```

Keep it proportionate — the goal is *traceability*, not a second copy of the file.
Keep a small replaced snippet verbatim; for a large block, a one-line `was:`
summary of what it did is enough. A later task may remove an older commented-out
block once it is clearly obsolete — tag that removal with the new task ID.

## Code style

- Match the conventions already in the surrounding file.
- Comment the **why**, not the **what** — default to no comment unless a
  constraint, invariant, or non-obvious workaround needs explaining. (The
  task-driven change markers above are the deliberate exception — they record
  change history, not behaviour.)
- Don't add abstractions, error handling, or backwards-compatibility shims beyond
  what the change requires.
- Keep secrets out of the repo and out of logs (`LogStore.redactSecrets()`
  handles redaction of keys and URI key-segments).

## Automated contributors

These conventions apply to AI assistants and agents too. Agents have a single
dedicated entry point with the operational workflow — task processing, versioning
under concurrency, build/test commands, change markers, and the hard don'ts: see
**[AGENTS.md](AGENTS.md)**, the source of truth for agent work. Keep it and this
file consistent.
